Paths And State

X4 stores repo-owned configuration in the registry and per-workspace state under the entry repo's .x4/ directory.

The short rule:

  • read .x4/context.md first when you are a human or agent;
  • read .x4/workspace.yaml when you need current machine-readable state;
  • read .x4/workspace.lock.yaml when you need latest observed repo SHAs, dirty flags, sandbox state, preview state, or test artifact pointers.

Generated .x4/ files are local workspace state. They are intentionally untracked. X4 adds .x4/ to local Git excludes instead of editing the repo's tracked .gitignore.

Source Paths​

PathMeaning
src/cli.tsRoot CLI source.
dist/cli.jsBuilt CLI binary target.
documentation/Docusaurus docs site.
docs/Existing specs and historical docs.
apps/mastra-orchestrator/Mastra app source when the current checkout includes it; some X4 checkouts only contain the docs, mirror workflow, and integration contracts.

Workspace Files​

Bootstrap writes these files under the entry repo:

.x4/
  context.md
  commands.md
  agent-instructions.md
  workspace.yaml
  workspace.lock.yaml
  x4-workspace.code-workspace
  repos/<attached-repo>/
PathMeaning
.x4/context.mdHuman/agent summary generated from the workspace and registry. Lists repo paths, roles, branch, setup commands, test commands, sandbox profile, and common X4 commands.
.x4/commands.mdGenerated command cheatsheet for workspace commands and per-repo tests.
.x4/agent-instructions.mdShort agent rules: read context first, use workspace metadata, avoid secrets, avoid public preview URLs.
.x4/workspace.yamlCurrent machine-readable workspace state. Records repo set, registry path, selected branch, entry repo, attached repo paths, sandbox state, preview state, worker target, ports, and runtime env references.
.x4/workspace.lock.yamlLatest observed state. Records repo HEAD SHAs, dirty flags, sandbox state, preview state, and latest test run pointer. Rewritten by status-changing commands.
.x4/x4-workspace.code-workspaceCursor/VS Code multi-root workspace file. Open this file to load all repos together.
.x4/repos/<attached-repo>/Attached repo worktrees/checkouts. The entry repo is the workspace root; non-entry repos live here.

context.md is deterministic, not AI-generated. It is rendered from workspace.yaml plus the registry when X4 writes workspace files.

Sandbox State​

x4 sandbox up writes sandbox state into .x4/workspace.yaml and .x4/workspace.lock.yaml.

For local service sandboxes, .x4/sandbox/docker-compose.yaml is the generated Compose file and Docker containers are labeled with:

com.x4.workspace-id=<workspace id>
com.x4.repo-set=<repo set>

x4 sandbox status uses those labels to reconcile saved workspace state with the actual Docker runtime. If .x4/workspace.yaml says a service sandbox is running but Docker has no running X4-labeled containers for that workspace, X4 marks the sandbox and preview stopped before printing status. Service and preview URLs are hidden in the stopped status output so dead links are not shown as tester links.

For remote worker sandboxes, the local workspace records:

  • remote worker id and SSH user;
  • remote workspace path;
  • remote artifact path;
  • compose project name;
  • local copied SSH logs under .x4/sandboxes/<id>/.

Cleanup Commands​

Use the narrowest cleanup command that matches the job:

CommandScope
x4 sandbox downStop the current workspace sandbox but keep state.
x4 sandbox destroy --force --prune-artifactsDestroy the current workspace sandbox and remove generated sandbox files.
x4 sandbox clear --all --dry-runList X4-labeled sandbox resources on the current Docker host.
x4 sandbox clear --all --forceRemove discovered X4-labeled sandbox containers, Compose networks, Compose volumes, and X4 preview proxy processes on the current Docker host.
x4 sandbox clear --target <worker> --all --dry-runList X4-labeled sandbox resources on a Docker-capable worker.
x4 sandbox clear --target <worker> --all --forceClear X4-labeled sandbox resources on a Docker-capable worker.
x4 cleanup --dry-runShow generated attached worktrees, sandbox state, preview state, and artifacts that would be cleaned for the workspace.
x4 cleanup --force --prune-artifactsRemove generated attached worktrees and generated X4 artifacts for the workspace.

sandbox clear is host-level cleanup. It is intentionally guarded by --all and --force, and it only targets resources with X4 labels or X4 preview proxy process paths. Use --repo-set <id> or --workspace-id <id> to narrow it.

What Agents Should Read​

When an agent opens a workspace:

  1. Read .x4/context.md.
  2. Read .x4/agent-instructions.md.
  3. Use .x4/workspace.yaml for current repo paths and sandbox/preview state.
  4. Use .x4/workspace.lock.yaml when reporting exact repo SHAs or dirty state.
  5. Do not commit .x4/ files unless a maintainer explicitly asks for a generated artifact.
